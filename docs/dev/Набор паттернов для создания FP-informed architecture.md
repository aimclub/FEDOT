# Паттерны которые нужно обязательно внедрять

#### 1) Pure core + thin effect shell

Это самый ценный паттерн из всего FP-наследия. Логика вычислений должна жить в чистых функциях, а доступ к файлам, сети, БД, логам, GPU-runtime и orchestration — на внешнем слое. 

**Где применять для наших репо:**
- препроцессинг датасетов;
- создание признаков или их преобразование;
- семплирование;
- агрегация метрик;
- planning/scheduling;
- проверка и нормализация конфигов.

**Почему это даст эффект:**
- тесты становятся короткими и детерминированными;
- проще параллелить и кэшировать;
- легче отделить исследовательский код от framework API.

**Критичное замечание** - не надо пытаться сделать чистыми функциями вообще всё. Инициализация CUDA, файловый streaming, распределенные лаунчеры, логгеры — это естественные эффекты запуска для нашего ПО. Цель не убрать их, а локализовать.

#### 2) Immutable domain model + algebraic data types

В функциональной парадигме есть сильный акцент на иммутабельных типах данных и ADT. В Scala 3 это реализовано через enums/sealed hierarchies. Это особенно полезно там, где сейчас много “stringly-typed orchestration”: статусы задач, типы бэкендов, типы стратегий семплирования, режимы inference/training/export.

**Что заменить:**
- флаги вида `mode = "train" | "infer" | "export"`
- nullable state objects
- dict-like config blobs

**На что заменить:**
- `case class` для стабильных структур данных;
- `enum` / `sealed trait` для закрытых наборов состояний;
- отдельные типы для domain events / commands / results.

**Практическая ценность для ML-кода:**
- "Состояния" мл-пайплайнов (pipeline states) становятся исчерпывающими;
- Проверка через match-компилятор ловит пропущенные кейсы;

#### 3) Typed error handling вместо null/exceptions как default-пути

Идея в том чтобы строить обработку ошибок без exceptions через `Option` и `Either`, `None` вместо `null`, и `Option/Try/Either` вместо бросания исключений для ожидаемых ошибок.

**Где это особенно полезно:**

- разбор и валидация конфигов;
- загрузка чекпоинтов;
- разрешение проблем с выбором "бекенда" под задачу;
- Обработка случаешь для feature extraction, где признак может отсутствовать;
- Семплирование датасетов, где часть источников может быть пустой/невалидной.

**Рекомендация:**
- `Option[A]` — когда отсутствующее значение нормально и без объяснений;
- `Either[DomainError, A]` — когда важен типизированный отказ;

**Критичное замечание** - **НЕ НАДО ТАЩИТЬ Either** везде, где достаточно `Option`. И не надо использовать `Try` как “универсальный эффект”.  Для "доменных ошибок** `Either` обычно лучше.

#### 4) Expression-oriented style

В ФП стоит мыслить выражениями, а не процедурами. `if`, `match`, `for`, `try` — всё это должно **ВОЗВРАЩАТЬ** значения, а не управлять скрытым изменяемы состоянием (бесконечные цепочки if-ов которые нереально дебажить). Это резко улучшает читаемость и композицию.

**Почему это важно в research/ML code:**

- меньше промежуточных изменяемых состояний;
- проще рефакторить длинные циклы preprocess/train/eval pipelines;
- проще разбивать "огромные методы" на композицию небольших функций.

**Антипаттерны:**
- Огромные методы с 20-30 переменными;
- “prepare + patch + override + postprocess” через неявные побочные эффекты.

#### 5) Traits/modules для модульности и environment-specific wiring

Traits как modules - это прямой способ делать dev/test/prod wiring, mock backends и.т.д.

**Где применять:**
- слой хранения данных: local fs / S3 / HDFS / artifact store;
- слой исполнения: CPU / single GPU / distributed / remote service;
- слой "репозитория" моделей;
- слой загрузки и обработки датасетов;

**Рекомендация:**
- бизнес-логика не должна знать, где именно лежит чекпоинт;
- orchestration depends on traits/interfaces;

**Критичное замечание:** - не злоупотреблять trait-mixins ради “умности”. Если обычный constructor DI проще — использовать его.

#### 6) Property-based testing и laws

FP in Scala уделяет отдельную главу property-based testing и для framework-кода это особенно ценно - там чаще ломаются не отдельные примеры, а **инварианты**.

**Где property tests особенно окупаются:**
- семплеры;
- schedulers;
- partitioners/shard assignment;
- tensor/data transforms;
- config merge/override;
- serialization round-trips;

**Примеры свойств:**
- идемпотентность normalization;
- `deserialize(serialize(x)) == x`;
- проверка что семплер учитывает "границы" выборки и seed determinism;
- процесс батчинга корректно сохраняет общее число семплов;

**Критичное замечание** - Property-based tests не заменяют integration tests. Они отлично ловят "баги в алгебре вычислений", но плохо ловят реальные issues.

#### 7) Local mutation behind a pure boundary

Это очень важная мысль из FP, которую часто не понимают либо фанатики, либо скептики. Книга прямо показывает, что локальная мутация внутри функции может быть допустима, если она не протекает наружу и не ломает внешнего API.

**Для НАШИХ ПРОЕКТОВ ЭТО АРХИВАЖНО** потому что иногда разумно писать императивно.:

- Тензорные ядра;
- Парсеры;
- in-place transformations для производительности

**Правило:**

- снаружи — чистый, детерминированный API;
- внутри — разрешена локальная мутация, если она изолирована и измеримо полезна.

**Это один из немногих FP-паттернов, который реально совместим с high-performance ML code.**

## Внедрять выборочно

#### 8) Typeclasses / givens / extension methods

Для framework-дизайна это очень полезно, когда надо расширять поведение без взрыва наследования.

**Хорошие применения:**
- serialization/deserialization;
- metric aggregation;
- backend-specific execution;
- numeric / tensor-like abstractions.

**Где не надо:**
- не вводить typeclass, если у вас один backend и два метода;
- не делать typeclass там, где простой trait field injection понятнее.
    
Рекомендация - осваивать обязательно. Массово тащить в код — только там, где правда нужна extensibility across types.

#### 9) Monoid / semigroup-мышление для aggregation

FP in Scala выводит моноиды как общую схему комбинирования и отдельно связывает их с parallelism. Для наших проектов это очень практично, хотя обычно недооценивается.

**Где применять:**

- агрегация метрик по shard/worker/run;
- мердж статистик датасета;
- объединение warnings/errors;

**Примерно так думать:**
- есть нейтральный элемент?
- есть associative combine?
- можно ли агрегировать частично и потом сводить?

**Критичное замечание** - не нужно учить абстрактную алгебру ради самой алгебры. Нужна практическая привычка: “что у меня за операция объединения, ассоциативна ли она, можно ли ее распараллелить?”

#### 10) Monad / for-comprehension workflows. Applicative / Traverse для валидации и обхода структур

FP in Scala раскрывает monads/applicatives/traverse уже как общие схемы композиции. Это полезно, но здесь легче всего скатиться в переусложнение. FP in Scala подчеркивает отличие applicative от monad и полезность traverse. Это очень полезно для config trees, batch initialization и массовой валидации.

**Где реально полезно:**

- многошаговые пайплайны с зависимыми шагами;
- error-aware orchestration;
- resource-aware sequential composition.
- валидация сложного train config;
- запуск набора независимых проверок;
- обход дерева pipeline-nodes;
- transform of nested specs.

**Где вредно:**
- тащить monad-термины в простой CRUD-like orchestration;
- заворачивать весь код в nested transformers без острой нужды.
