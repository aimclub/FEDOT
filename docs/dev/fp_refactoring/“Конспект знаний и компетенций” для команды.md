
### Модуль 1. FP-мышление без догмы

Разработчик должен понимать:

- что такое pure function;
- что такое referential transparency;
- почму side effects ухудшают local reasoning;
- почему pure code проще тестировать и распараллеливать

**Практический результат:**  
Умеет отличить pure core от effectful shell в существующем модуле.


### Модуль 2. Моделирование домена

Разработчик должен уметь:

- моделировать данные через case classes;
- моделировать закрытые наборы состояний через enums / sealed ADTs;
- избегать `Map[String, Any]` и sentinel values;
- использовать pattern matching как основной инструмент разбора состояний.

**Практический результат:**  
Умеет описать train job, export plan, backend capability set, routing outcome как typed domain model.


### Модуль 3. Ошибки как данные

Разработчик должен уметь:

- выбирать между `Option`, `Either`, `Try`;
- не использовать `null`;
- не бросать exceptions для ожидаемых ошибок;
- моделировать domain errors явно.

**Практический результат:**  
Умеет переписать fragile config/parser API в typed validation API.

### Модуль 4. Композиция через expressions и combinators

Разработчик должен уметь:

- писать код как цепочки выражений;
- использовать `map`, `flatMap`, `filter`, `fold`, `collect`, `traverse`-подобное мышление;
- убирать ручные циклы там, где декларативный обход понятнее.

**Практический результат:**  
Умеет превратить длинный императивный метод препроцессинга в композицию небольших трансформаций.

### Модуль 5. Модульность и расширяемость

Разработчик должен уметь:
- выделять trait/interface границы;
- строить modules через traits + objects/DI;
- не смешивать domain logic с env wiring.

**Практический результат:**  
Умеет сделать заменяемый storage/runtime/model-registry бекенд

### Модуль 6. Тестирование инвариантов

Разработчик должен уметь:

- писать unit тесты для чистых функций;
- проверять deterministic behavior;
- делать round-trip / associativity / idempotence / monotonicity tests там, где это уместно.

### Модуль 7. State, effects, resources

Разработчик должен понимать:

- что state можно моделировать явно;
- что effects можно вытеснять к границам;
- что локальная мутация внутри pure boundary допустима;
- что resource lifecycle нельзя прятать в “магические helper-и”.


**Практический результат:**  
Умеет спроектировать API, которое снаружи детерминировано, а внутри допускает буферы, in-place updates и streaming.

### Модуль 8. FP для performance-sensitive systems

Разработчик должен уметь различать:

- где иммутабельность это залог безопасности;
- где локальная мутация допустима и может дать прирост в производительности;
- где абстракции не имеют смысла или дают мало пользы;

**Ключевая компетенция здесь — не “быть чистым”, а уметь ставить правильную границу.**

## 3. Код-ревью чеклист

1. Можно ли вынести вычислительную логику в чистую функцию?
2. Не скрыта ли ошибка через `null` / unchecked exception?
3. Можно ли сделать domain state исчерпывающим через enum / ADT?
4. Не мутируется ли shared state без крайней необходимости?
5. Можно ли упростить код через выражения и композиции?
6. Если используется мутация — локальна ли она и оправдана ли профилированием?
7. Не введён ли слишком “умный” уровень абстракции просто так?
